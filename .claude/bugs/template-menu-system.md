# Bug Report: 网页模板制作和菜单关联系统问题

## 问题描述
网页模板制作和菜单关联功能实现混乱，用户体验差，代码结构不清晰。

## 当前实现分析

### 数据模型
- **Menu 实体**：支持关联 page_id、article_id、category_id
- **Page 实体**：有 layout_config (JSON) 和 template (字符串) 字段
- **模板文件**：在 `admin/src/templates/` 下有 7 个预设模板

### 存在的问题

#### 1. 模板系统混乱
- ❌ Page 实体同时有 `layout_config` 和 `template` 字段，职责不清
- ❌ 模板是硬编码的 TypeScript 文件，不灵活
- ❌ 没有模板管理界面，用户无法自定义模板
- ❌ 模板和页面的关系不明确

#### 2. 菜单关联复杂
- ❌ Menu 可以关联 page/article/category，逻辑复杂
- ❌ 前端路由和菜单关联不清晰
- ❌ 菜单创建时需要手动填写 path，容易出错

#### 3. 用户体验差
- ❌ 创建页面流程不清晰：先创建页面？还是先选模板？
- ❌ 菜单关联页面时，不知道页面用的什么模板
- ❌ 修改模板后，关联的菜单不会自动更新

## 期望的实现

### 理想流程
1. **选择模板** → 从模板库选择（首页/关于/产品/博客等）
2. **创建页面** → 基于模板创建，可自定义内容
3. **创建菜单** → 选择页面，自动关联路径
4. **发布** → 前端自动渲染

### 核心改进点
- ✅ 模板和页面分离：模板是"蓝图"，页面是"实例"
- ✅ 简化菜单关联：只关联 page_id，路径自动生成
- ✅ 可视化编辑：拖拽组件，实时预览
- ✅ 模板管理：可以保存自定义模板

## 技术债务
- 数据库字段冗余（layout_config vs template）
- 前端组件类型定义不完整
- 缺少模板版本管理
- 缺少页面历史记录

## 影响范围
- 后端：Menu、Page 实体和 API
- 前端管理后台：页面编辑器、菜单管理
- 前端官网：路由和页面渲染

## 优先级
🔴 **高优先级** - 这是核心功能，直接影响系统可用性

## 建议方案
需要重新设计模板-页面-菜单的关系模型，分阶段重构。
