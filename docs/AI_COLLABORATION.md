# AI 协作指南

## 🤖 如何与 AI 协作开发

本文档指导如何使用 Claude Code 高效完成开发任务。

## 📋 角色定义

### 需求分析师 (Demand Analyst)
**触发命令**：`/analyze-demand`

**职责**：
- 分析用户需求，拆解为可执行任务
- 评估技术可行性和风险
- 制定优先级和时间计划

**输出**：
- 需求分析报告
- 任务清单（带优先级）
- 风险评估文档

**使用示例**：
```bash
/analyze-demand 我想添加一个用户评论功能，支持点赞和回复
```

## 🔄 标准工作流程

### 1. 需求阶段
```bash
# 步骤1：需求分析
/analyze-demand [需求描述]

# 步骤2：架构设计
/design-arch [具体功能]

# 步骤3：确认技术方案
# AI 会输出设计文档，人工审核确认
```

### 2. 开发阶段
```bash
# 步骤1：前端开发
/01-dev 实现 [功能名称] 的前端界面

# 步骤2：后端开发
/01-dev 实现 [功能名称] 的后端API

# 步骤3：联调测试
/02-test 测试 [功能名称]
```

### 3. 审查阶段
```bash
# 步骤1：代码审查
/code-review 审查 [文件路径]

# 步骤2：安全审查
/05-security 审查 [模块名称]

# 步骤3：修复问题
/01-dev 修复 [问题描述]
```

### 4. 部署阶段
```bash
# 步骤1：配置部署
/03-deploy 配置 [部署环境]

# 步骤2：执行部署
# 按照 AI 输出的部署文档执行

# 步骤3：验证部署
/02-test 验证生产环境功能
```

## 💡 最佳实践

### 1. 任务拆解原则
- **单一职责**：每个任务只做一件事
- **可测试**：任务完成后能够验证
- **有依赖**：明确任务之间的依赖关系
- **有时限**：预估合理的完成时间

### 2. 与 AI 沟通技巧
- **明确需求**：清晰描述要做什么，而不是怎么做
- **提供上下文**：告诉 AI 相关的文件路径和背景信息
- **分步执行**：复杂任务分多次对话完成
- **及时反馈**：AI 输出后及时确认或提出修改意见

### 3. 代码质量保证
- **先读后写**：修改代码前先让 AI 读取现有代码
- **小步提交**：每完成一个小功能就提交代码
- **测试驱动**：先写测试，再写实现
- **代码审查**：重要代码必须经过审查

### 4. 文档管理
- **及时更新**：代码变更后同步更新文档
- **版本控制**：重要文档纳入 Git 管理
- **清晰命名**：文档命名要见名知意
- **结构化**：使用统一的文档模板

## 🎯 任务模板

### 开发新功能
```markdown
## 功能：[功能名称]

### 需求描述
[详细描述功能需求]

### 技术方案
[简要说明技术实现方案]

### 任务清单
- [ ] 数据库设计
- [ ] 后端API开发
- [ ] 前端界面开发
- [ ] 单元测试
- [ ] 集成测试
- [ ] 文档更新

### 验收标准
- [ ] 功能正常运行
- [ ] 测试覆盖率 > 80%
- [ ] 代码审查通过
- [ ] 文档已更新
```

### 修复 Bug
```markdown
## Bug：[Bug 描述]

### 问题现象
[详细描述 Bug 表现]

### 复现步骤
1. [步骤1]
2. [步骤2]
3. [步骤3]

### 预期行为
[描述正确的行为]

### 修复方案
[说明修复思路]

### 验证方法
[如何验证 Bug 已修复]
```

## 🚨 注意事项

### 安全相关
- ❌ 不要在代码中硬编码敏感信息
- ❌ 不要提交 .env 文件到仓库
- ❌ 不要在日志中输出用户隐私数据
- ✅ 所有输入必须验证和消毒
- ✅ 数据库查询必须使用参数化
- ✅ API 必须有速率限制

### 代码规范
- ✅ 所有代码必须有类型注解
- ✅ 函数不超过 50 行
- ✅ 类不超过 300 行
- ✅ 禁止使用 any 类型（特殊情况需注释）
- ✅ 公共 API 必须有文档注释

### Git 规范
- ✅ 提交信息格式：`<type>(<scope>): <description>`
- ✅ 类型：feat、fix、docs、style、refactor、test、chore
- ✅ 每次提交只做一件事
- ✅ 提交前运行测试
